# 全自动书籍带货视频工作流设计

## 目标

在现有 `book-video` 工作流上增加一个统一的全自动生产入口。用户可以指定一本书、要求系统自动选书，或提交批量书单。系统完成书籍研究、文案生成、AI 生图、剪映配音、字幕时间轴、视频渲染和技术验收，最终交付本地 MP4。

第一版不自动发布到抖音、小红书或视频号，也不管理商品链接。生产边界到经过技术验收的本地视频为止。

## 核心决策

- 仓库负责选书、研究、文案、生图、时间轴、画面渲染、混音和验收。
- 剪映只作为文字转语音引擎，不承担视频剪辑。
- 全自动模式跳过人工文案和图片审核，但保留所有机器质量门。
- 批量模式逐本执行，避免多个任务争抢剪映窗口。
- 每个阶段完成后保存状态，失败任务从失败阶段继续，不重复已完成工作。
- `script.csv` 始终是配音和字幕的唯一文本真源。

## 启动模式

统一入口支持以下模式：

```bash
node scripts/auto-produce.mjs book "我与地坛"
node scripts/auto-produce.mjs auto --theme "孤独与成长"
node scripts/auto-produce.mjs batch "活着" "悉达多" "局外人"
node scripts/auto-produce.mjs resume "我与地坛"
```

用户通常直接向 Agent 发出自然语言指令，由 Agent 转换为对应命令。

- `book`：验证指定书籍的书名、作者和版本后直接生产。
- `auto`：根据主题或默认受众从候选池选择一本尚未生产的书。
- `batch`：按输入顺序逐本生产；一本失败不阻止后续任务。
- `resume`：读取现有状态，从首个未完成阶段继续。

## 生产流程

```text
用户指令
  -> 识别启动模式
  -> 选书与书目信息确认
  -> 微信读书优先、公开资料补充的研究
  -> 生成并自动校验文案
  -> 生成三张氛围图和一张结果桥接图
  -> 自动操作剪映生成正文口播 MP3
  -> Whisper 和静音检测生成字幕时间轴
  -> HyperFrames 渲染片头与正文
  -> FFmpeg 拼接、混音和编码
  -> 技术与视觉验收
  -> 交付 MP4 和生产报告
```

配音文本必须从当前活动版本的 `script.csv` 构造。剪映配音包含书名；时间轴生成默认跳过开头的书名朗读段。ASR 只提供时间参考，字幕文本不使用 ASR 转录结果。

## 组件设计

### 总控入口

`scripts/auto-produce.mjs` 负责：

- 解析模式和参数；
- 创建或恢复生产任务；
- 顺序调用现有及新增阶段模块；
- 检查阶段前置条件和输出；
- 更新状态和最终报告；
- 在批量模式中隔离单本失败。

总控入口不重复实现文案校验、时间轴、预览和最终渲染逻辑，而是调用现有脚本。

### 剪映配音控制

剪映自动化负责：

- 打开或聚焦剪映；
- 处理已知登录、草稿恢复和提示弹窗；
- 创建临时配音草稿；
- 粘贴由 `script.csv` 生成的完整配音文本；
- 首次运行自动选择普通话、自然、克制的叙事音色；
- 后续运行优先复用已记录的音色；
- 导出 MP3 并等待文件写入完成；
- 验证导出文件不是上次任务遗留文件。

如果原音色不可用，系统重新选择符合条件的音色，并在报告中记录变更。界面识别持续失败时，任务停在配音阶段并保留前序产物。

### 现有组件复用

- `scripts/record-book-candidates.mjs`：保存和去重候选书。
- `scripts/validate-script.mjs`：文案长度和结构校验。
- `scripts/create-body-timings.mjs`：Whisper、静音检测和字幕对齐。
- `scripts/create-episode-preview.mjs`：构造 HyperFrames 片头和正文项目。
- `scripts/process-voiceover.mjs`：故事感口播处理。
- `scripts/render-episode-final.mjs`：画面渲染、混音、编码和最终文件激活。
- `scripts/check.mjs`：仓库、模板和运行时检查。

## 状态与文件

每本书沿用现有目录，并增加状态和报告：

```text
episodes/<书名>/
  brief.json
  script.csv
  prompts.csv
  production-state.json
  production-report.json
  images/
  audio/
  renders/
```

状态按以下顺序推进：

```text
selected -> researched -> scripted -> illustrated
         -> voiced -> timed -> rendered -> verified
```

`production-state.json` 至少记录当前阶段、活动文案版本、各阶段完成时间、失败阶段、失败原因和重试次数。`production-report.json` 记录最终音色、BGM、时长、媒体规格、输出路径和验收结果。

本地全局配置保存在被 Git 忽略的 `.book-video-config.json`，包括：

- 剪映应用和导出目录；
- 已选音色；
- 默认主题和受众；
- 默认 BGM 策略；
- 自动选书排除已生产书籍的规则；
- 各阶段超时和重试次数。

用户凭证、剪映账号数据、录音、生成图片、模型和渲染结果不得提交到 Git。

## 首次配置

首次运行执行一次引导式准备：

1. 检查 Node.js 22+、`npx`、FFmpeg、FFprobe、Whisper CLI 和 Whisper 模型。
2. 确认剪映已经安装并登录。
3. 创建临时配音草稿并测试候选音色。
4. 选择并记录符合目标风格的音色。
5. 导出测试 MP3，验证界面路径和导出目录。
6. 渲染短测试视频，确认画幅、字幕和音轨。
7. 保存本地配置，启用无人值守生产。

当前环境尚未找到 `node`，实施和实机验收前必须先完成 Node.js 22+ 及其他运行时检查。

## 异常处理与续跑

- 研究失败：切换到公开资料；仍无法可靠确认作者和版本时标记失败。
- 文案不合格：自动重写或缩短，最多三次，仍不通过则停止。
- 生图失败：只重试缺失图片，不重新研究或改写文案。
- 剪映操作失败：保存截图和错误信息并按配置重试；持续失败则停在 `illustrated` 阶段。
- 配音异常：检查格式、大小、时长、修改时间和任务开始时间，拒绝旧文件。
- 字幕对齐不完整：使用预设的备用静音阈值重新计算；仍不完整则禁止渲染。
- 渲染失败：保留临时项目和日志，从 `timed` 阶段续跑。
- 最终验收失败：候选成片不得进入正式 `renders/`，也不得覆盖已有有效成片。

批量任务为每本书生成独立结果，最终汇总成功、失败、失败阶段和续跑建议。

## 质量门

最终视频必须满足：

- 分辨率为 `720x960`，帧率为 30fps；
- 使用 H.264 视频和 AAC 音频的 MP4；
- 默认时长不超过 60 秒；
- 存在一张结果桥接图和三张氛围图；
- 字幕数量与当前 `script.csv` 的正文行数完全一致；
- 片头口播、正文口播、BGM 和齿轮音效均存在；
- 无空白帧、占位文本和明显字幕越界；
- 技术验收通过后才替换当前活动成片。

文案仍遵循现有仓库规则：推荐连书名共 18 至 20 行，硬上限 22 行；正文约 220 个汉字，不使用 CTA、机械排比或生硬书评式表达。

## 测试策略

### 单元测试

- 三种模式和 `resume` 参数解析；
- 状态机合法流转和失败恢复；
- 自动选书排重；
- 配音文本从 `script.csv` 构造；
- 导出文件新旧识别；
- 生产报告生成。

### 集成测试

使用模拟的剪映 MP3 导出文件跑通文案、生图占位、时间轴、渲染和验收流程，验证总控逻辑不依赖真实界面才能测试。

### 实机验收

选择一本书，真实操作剪映完成音色选择和 MP3 导出，再生成最终 MP4。随后中断一次任务并执行 `resume`，验证不会重复已完成阶段。最后以至少三本书执行批量任务，验证单本失败隔离。

## 成功标准

- 四种命令入口均可运行，其中三种生产模式交付最终 MP4；
- 首次配置完成后，单本生产无需人工确认；
- 同一书籍失败后可以从失败阶段继续；
- 批量任务不会因单本失败整体终止；
- 所有正式成片通过现有和新增质量门；
- 最终回复直接嵌入本地视频，并附生产报告摘要。
